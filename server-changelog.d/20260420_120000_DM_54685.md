### New features

- Tracer-bullet dashboard render pipeline: a new `dashboard_build` arq worker function renders a project's dashboard HTML and pydata-sphinx-theme switcher JSON from a packaged built-in Jinja template and uploads them to the project's object store. Operators can trigger a render manually via `POST /orgs/{org}/projects/{project}/dashboard/rebuild`.
- New `JobKind.dashboard_build` queue value (distinct from the still-reserved `dashboard_sync`) so the MVP render pipeline and the future template-sync pipeline are tracked as separate queue concepts.
- New `editions.alternate_name` column propagated through the `Edition` domain model and dashboard `EditionContext` so deployment-scoped tracking rules can surface their target into templates.
- New `Configuration.published_base_url` setting, surfaced into dashboard templates as `docverse.api_url`. Defaults to `https://roundtable-dev.lsst.cloud/docverse/api` until the Repertoire integration lands.
- New `AssetInliner` service inlines the CSS, JS, and image assets declared in a template's `template.toml` into the dashboard `AssetsContext`. CSS and JS are concatenated in declared order; SVG images inline as raw markup; PNG/JPEG/GIF/WEBP images encode as base64 `data:` URIs. Image keys use the basename with dots and hyphens replaced by underscores (e.g. `favicon-32x32.png` → `favicon_32x32_png`).
- The packaged built-in dashboard template now ships a minimal CSS file, JS file, SVG logo, and PNG favicon so the asset-inlining pipeline is exercised end-to-end out of the box.
- New `ErrorPageRenderer` produces a `{project_slug}/__404.html` artifact alongside the dashboard and switcher uploads. When a template's `template.toml` declares `[error_404]`, the configured Jinja template is used with its own inlined assets; otherwise the renderer falls back to a packaged default 404 page that links back to the project dashboard.
- New `EditionJsonRenderer` produces one `{project_slug}/__editions/{edition_slug}.json` file per non-deleted edition on every dashboard render. Each file advertises `is_canonical` (true only for `__main`) and `canonical_url` (always the `__main` edition's `published_url`), so a change to `__main` propagates to every edition's metadata on the next render without per-edition bookkeeping.
